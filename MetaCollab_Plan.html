1. Project Overview

What is MetaCollab?

MetaCollab is a collaborative project management platform that enables teams of 5-20 people to work together. It combines project management, real-time task tracking, and instant messaging in one intuitive interface.

Core Features (MVP)

  • User authentication via Clerk
  • Create and manage projects
  • Invite team members by email
  • Kanban board with real-time updates
  • Ephemeral live chat (messages vanish when workspace is empty)
  • Real-time presence (see who's online)
  • Role-based access (Admin/Member)

Tech Stack (Final)

Frontend

Next.js 14+ with TypeScript

Backend

Next.js API Routes

Database

PostgreSQL + Prisma ORM

Authentication

Clerk (signup/login only)

Real-time

HTTP Polling (no WebSockets)

UI

Shadcn/ui + Tailwind CSS

2. Real-time Architecture (CORRECTED)

❌ WRONG Explanation

Do NOT do this: "Send message → Broadcast to all in workspace"
You are NOT using WebSockets. There is NO broadcast layer.

✅ CORRECT Flow: HTTP Polling

Live Chat Flow

User A sends message: 1. Type message → Click send 2. POST /api/projects/[id]/chat 3. Backend: Store message in DB 4. Return success to User A 5. Message appears immediately on User A's screen (optimistic update) User B receives message: 1. Browser runs: GET /api/projects/[id]/chat/poll?since=lastTimestamp 2. Poll runs every 2 seconds (NOT 1.5s - too aggressive) 3. Backend returns: { messages: [...], activeUsers: [...] } 4. User B's frontend updates chat display 5. Both see message in sync

Kanban Board Real-time Flow

User A drags task: 1. User A drags task "Design homepage" from Todo → In Progress 2. Optimistic update: Show change on User A's screen immediately 3. PUT /api/projects/[id]/tasks/[taskId] Body: { status: "IN_PROGRESS", order: 2 } 4. Backend: Update DB 5. Return success User B sees the update: 1. Browser runs: GET /api/projects/[id]/tasks 2. Poll runs every 4 seconds (slower than chat) 3. Backend returns full task list 4. Detect differences from local state 5. User B's board updates to show task in new position

⚠️ CRITICAL: Ephemeral Chat Cleanup Logic

WRONG: "Delete messages when last user leaves workspace"
Why it breaks: Browsers close unexpectedly, no reliable "leave" event

CORRECT: Use heartbeat timeout

How Cleanup Actually Works

ActiveUsers table tracks presence: - id, projectId, userId, lastHeartbeat User joins workspace: 1. POST /api/projects/[id]/chat/heartbeat 2. Create/update ActiveUsers row 3. Set lastHeartbeat = NOW() User stays in workspace: 1. Every 30 seconds: POST heartbeat again 2. Update lastHeartbeat = NOW() Cleanup job (run every 2 minutes): 1. Check all projects 2. Find where: lastHeartbeat < NOW() - 5 minutes 3. If NO active users for 5+ minutes: → DELETE all ChatMessages for that project → DELETE all ActiveUsers entries → Workspace becomes clean slate

Polling Intervals (Optimized)

What Interval Reason
Chat messages 2000ms (2 seconds) Feels instant, ~3 req/sec per user, manageable
Task list (Kanban) 4000ms (4 seconds) Less frequent changes, reduce load
Active users 2000ms (with chat poll) Included in chat response, no extra request
Heartbeat 30000ms (30 seconds) Keep user marked as active (prevents cleanup)

Load Estimation (5-20 users)

With 10 users in a workspace:
• Chat poll: 10 users × 1 req/2s = 5 req/sec
• Task poll: 10 users × 1 req/4s = 2.5 req/sec
• Total: ~7.5 req/sec = manageable, no scaling issues

3. Database Schema with Indexes

Prisma Schema (schema.prisma)

Users Table

id String (PK) Clerk user ID (clerk_xxx) - source of truth
email String Email (cached from Clerk for reference)
name String Full name
createdAt DateTime Account creation
⚠️ SECURITY: Never trust email/name from client. Always read from Clerk session. This table is for queries only - user identity comes from Clerk.

Projects Table

id String (PK) UUID
name String Project name
description String (nullable) Optional description
adminId String (FK) Creator (always admin)
createdAt DateTime Creation timestamp

ProjectMembers Table

id String (PK) UUID
projectId String (FK) Which project
userId String (FK) Which user
role Enum "ADMIN" or "MEMBER"
joinedAt DateTime When user joined
Unique constraint @@unique([projectId, userId]) No duplicate memberships
Indexes @@index([userId]), @@index([projectId]) Fast lookups by user or project

ProjectInvitations Table

id String (PK) UUID
projectId String (FK) Which project
email String Email being invited
invitedByUserId String (FK) Who sent the invitation
status Enum "PENDING", "ACCEPTED", "DECLINED"
createdAt DateTime Invitation timestamp
respondedAt DateTime (nullable) When user accepted/declined

Tasks Table

id String (PK) UUID
projectId String (FK) Which project
title String Task name
description String (nullable) Task details
status Enum "TODO", "IN_PROGRESS", "DONE"
order Int Position within status (see note below)
assignedToUserId String (FK, nullable) Who it's assigned to
createdByUserId String (FK) Who created it
createdAt, updatedAt DateTime Timestamps
Compound Index @@index([projectId, status, order]) CRITICAL - for Kanban queries
Task Ordering Fix: "order" field alone will break. Always query with: ORDER BY status, order. Without compound index, Kanban becomes slow with 100+ tasks.

ChatMessages Table

id String (PK) UUID
projectId String (FK) Which workspace
userId String (FK) Who sent message
message String Message content
createdAt DateTime Send timestamp
Index @@index([projectId]), @@index([createdAt]) For polling queries and cleanup

ActiveUsers Table

id String (PK) UUID
projectId String (FK) Which workspace
userId String (FK) Which user
lastHeartbeat DateTime Last seen timestamp
Unique constraint @@unique([projectId, userId]) One entry per user per project
Index @@index([projectId]) For cleanup queries

4. API Endpoints & Security

⚠️ CRITICAL Security Rule

When fetching a project, ALWAYS verify user is a member first!

❌ WRONG:
    db.project.findUnique({ where: { id } })

✅ CORRECT:
    db.projectMembers.findFirst({
      where: { projectId: id, userId }
    })

Otherwise, users can guess project IDs and see private workspaces.

Projects API

GET /api/projects?limit=20&cursor=...

List projects user is member of. Pagination required for scalability. Returns: array of projects.

POST /api/projects

Create project. Body: {name, description}. User becomes admin. Returns: created project.

GET /api/projects/[id]

MUST verify membership first. Returns: project + members.

DELETE /api/projects/[id]

Delete project (admin only). Verify membership + admin role.

Tasks API

GET /api/projects/[id]/tasks

Query with: ORDER BY status, order. Verify membership.

POST /api/projects/[id]/tasks

Create task. Body: {title, description, assignedToUserId}. Verify membership.

PUT /api/projects/[id]/tasks/[taskId]

Update task. Body: {title, description, status, order, assignedToUserId}. Creator or admin only.

DELETE /api/projects/[id]/tasks/[taskId]

Delete task (creator or admin).

Invitations API (REVISED)

POST /api/projects/[id]/invitations

Send invitation. Body: {email}. Admin only.

GET /api/invitations/pending

Get pending invitations for current user.

PATCH /api/invitations/[id]

CLEANER API: Body: {status: "ACCEPTED" | "DECLINED"}. Updates invitation status and creates ProjectMember if accepted.

Chat API (with Polling)

POST /api/projects/[id]/chat

Send message. Body: {message}. Verify membership. Returns: message object.

GET /api/projects/[id]/chat/poll?since=ISO_TIMESTAMP

Poll for new data: Query param "since" to only get messages after timestamp. Returns: {messages, activeUsers, timestamp}.

POST /api/projects/[id]/chat/heartbeat

Keep user marked as active. Updates/creates ActiveUser entry. Every 30 seconds from client.

Members API

GET /api/projects/[id]/members

List project members. Verify membership.

DELETE /api/projects/[id]/members/[userId]

Remove member (admin only).

Two-Layer Permission System

Layer 1 - Authentication (Clerk): "Who are you?" - Clerk handles
Layer 2 - Authorization (Database): "What can you do?" - ProjectMembers table handles

Permission Checks (Backend)

// lib/auth.ts export async function getAuth() { const { userId } = auth() // From Clerk if (!userId) throw new Error('Unauthorized') return userId } export async function isProjectMember(projectId: string, userId: string) { return db.projectMembers.findFirst({ where: { projectId, userId } }) } export async function isProjectAdmin(projectId: string, userId: string) { return db.projectMembers.findFirst({ where: { projectId, userId, role: 'ADMIN' } }) } // Usage in API route: export async function DELETE(req, { params }) { const userId = await getAuth() // Security check: user must be admin const isAdmin = await isProjectAdmin(params.id, userId) if (!isAdmin) { return Response.json({ error: 'Forbidden' }, { status: 403 }) } // Safe to delete }

6. Service Layer Architecture

Why Services?

Keep API routes thin. Business logic goes in services.

Folder Structure

lib/services/ ├── project.service.ts (create, delete, list, getMembers) ├── task.service.ts (create, update, delete, list) ├── invitation.service.ts (send, accept, decline, list) ├── chat.service.ts (send, poll, cleanup) └── auth.service.ts (verify membership, verify admin)

Example Service

// lib/services/task.service.ts import { db } from '@/lib/db' export async function getProjectTasks( projectId: string, userId: string ) { // Verify membership await verifyProjectMembership(projectId, userId) // Get tasks ordered correctly return db.task.findMany({ where: { projectId }, orderBy: [ { status: 'asc' }, { order: 'asc' } ] }) } export async function updateTask( projectId: string, taskId: string, userId: string, data: any ) { // Verify membership await verifyProjectMembership(projectId, userId) // Check if user is creator or admin const task = await db.task.findUnique({ where: { id: taskId } }) const isAdmin = await isProjectAdmin(projectId, userId) if (task.createdByUserId !== userId && !isAdmin) { throw new Error('Unauthorized') } // Safe to update return db.task.update({ where: { id: taskId }, data }) } async function verifyProjectMembership(projectId: string, userId: string) { const member = await db.projectMember.findFirst({ where: { projectId, userId } }) if (!member) throw new Error('Not a member') } async function isProjectAdmin(projectId: string, userId: string) { const member = await db.projectMember.findFirst({ where: { projectId, userId, role: 'ADMIN' } }) return !!member }

Example API Route

// app/api/projects/[id]/tasks/[taskId]/route.ts import { auth } from '@clerk/nextjs' import { updateTask } from '@/lib/services/task.service' export async function PUT(req, { params }) { try { const { userId } = auth() if (!userId) return Response.json({ error: 'Unauthorized' }, { status: 401 }) const data = await req.json() const task = await updateTask( params.id, params.taskId, userId, data ) return Response.json(task) } catch (error) { return Response.json( { error: error.message }, { status: 403 } ) } }

Benefit: API routes stay clean. All security checks and business logic in one place. Easy to test.

7. Frontend Architecture

Polling Hooks

useChat Hook

// hooks/useChat.ts export function useChat(projectId: string) { const [messages, setMessages] = useState([]) const [activeUsers, setActiveUsers] = useState([]) const [lastPollTime, setLastPollTime] = useState(new Date()) // Poll every 2 seconds useEffect(() => { const interval = setInterval(async () => { const response = await fetch( `/api/projects/${projectId}/chat/poll?since=${lastPollTime.toISOString()}` ) const data = await response.json() if (data.messages.length > 0) { setMessages(prev => [...prev, ...data.messages]) } setActiveUsers(data.activeUsers) setLastPollTime(new Date(data.timestamp)) }, 2000) return () => clearInterval(interval) }, [projectId]) // Heartbeat every 30 seconds useEffect(() => { const interval = setInterval(() => { fetch(`/api/projects/${projectId}/chat/heartbeat`, { method: 'POST' }) }, 30000) return () => clearInterval(interval) }, [projectId]) return { messages, activeUsers } }

useKanban Hook

// hooks/useKanban.ts export function useKanban(projectId: string) { const [tasks, setTasks] = useState([]) // Poll every 4 seconds useEffect(() => { const poll = async () => { const response = await fetch(`/api/projects/${projectId}/tasks`) const data = await response.json() setTasks(data) } poll() // Initial fetch const interval = setInterval(poll, 4000) return () => clearInterval(interval) }, [projectId]) return tasks }
